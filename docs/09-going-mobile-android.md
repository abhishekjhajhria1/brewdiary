# 09 — Going mobile: Android (and iOS)

**Short answer: yes, absolutely — and we built the app to make it easy.** brewdiary is a web app, but a
web app can become a real phone app you install from the Play Store. You do *not* rewrite it. This
explains the three paths, what each costs, and which to pick. (Terms unclear? See the
[glossary](02-glossary.md).)

## The one prerequisite (matches your plan)
Every store-app path needs the website **deployed to a real internet address first** (e.g.
`https://brewdiary.app` on a host like Vercel). That's the "when everything on the website is ready to
push to users" moment. The phone app is a thin native shell around that live website — so the website
must be live first. (brewdiary uses server features — the AI route, login/session middleware — so it
can't be a pure offline bundle; it needs a server, which deploying provides.)

## What's already done (so the phone experience is good on day one)
The app is a proper **PWA** (Progressive Web App), and we just hardened it for phones:
- **Installable** — on Android, Chrome shows "Add to Home Screen"; it then opens fullscreen with its own
  icon, no browser bar. (Try it now on your phone against the dev/live site.)
- **Proper app icons** — real PNG icons at 192/512, a **maskable** icon (so Android's circular/rounded
  masks never clip the mark), and an Apple touch icon. Generated by `scripts/gen-icons.mjs`.
- **Notch-safe** — the layout respects **safe-area insets**, so the top bar and bottom tabs never hide
  under a phone's notch or home indicator.
- **Offline shell + dark app theme** on the splash/status bar, portrait-locked, correct manifest.

That means: even with **zero** extra work, users can install brewdiary on Android today and it feels
like an app. The paths below are about getting it into the **Play Store**.

## The three paths (pick based on what you need)

### Path A — PWA install only (free, done)
Users install straight from the browser ("Add to Home Screen"). No store, no fees, no build.
- ✅ Free, instant, one codebase, auto-updates when you deploy.
- ❌ Not in the Play Store (some users only trust the store); limited push notifications on iOS.
- **Use it:** as the baseline — it already works.

### Path B — TWA via Bubblewrap / PWABuilder (Play Store, minimal work) ⭐ fastest to store
A **Trusted Web Activity** wraps your live PWA into a real Android app package (`.aab`) for the Play
Store. Google's **Bubblewrap** CLI (or the **PWABuilder** website) generates it from your manifest.
- ✅ Real Play Store listing; tiny effort; stays one codebase; the app *is* your live site.
- ✅ No native code to maintain.
- ❌ Android only (for iOS you'd use one of the other paths). Push notifications are basic.
- **Needs:** the site deployed, a **Google Play Console** account ($25 one-time), a signing key, and a
  `.well-known/assetlinks.json` file on your site (proves you own the domain). Node only — no Android
  Studio required.
- **Use it:** if the near-term goal is simply "be in the Play Store" with least effort.

### Path C — Capacitor (native wrapper, Play **and** App Store) ⭐ best long-term
**Capacitor** (by the Ionic team) wraps the web app in a native shell and unlocks **native device
features** through plugins — while still reusing 100% of the Next.js code.
- ✅ Both **Android and iOS** from one codebase.
- ✅ **Native push notifications** — this is what makes the *nightly reminder* a real, reliable phone
  notification (a genuine feature win). Also native share (for the share-card), better camera, and
  geolocation (for Discover).
- ❌ A bit more setup; you install Android Studio / Xcode to build; you maintain a small native project.
- **Needs:** the site deployed (Capacitor points at the live URL), the Android SDK / Xcode, a Play
  Console ($25) and/or Apple Developer account ($99/yr), signing keys.
- **Use it:** when you want the reminder to buzz people's phones at night and to be on iOS too — i.e.
  the real product, not just a listing.

## Recommendation
1. **Now:** ship the PWA (Path A) — it's done. Let people install it while you finish the website.
2. **For the Play Store:** go **Capacitor (Path C)** rather than TWA, because brewdiary's roadmap needs
   **native push notifications** for the nightly-reminder habit loop (the whole point of the app), plus
   native share and iOS reach. TWA (Path B) is the right call *only* if you want a Play-Store listing
   this week with no native features yet — it's a fine stepping stone.
